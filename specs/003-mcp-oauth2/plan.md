# Implementation Plan: MCP OAuth2 User Authentication

**Branch**: `003-mcp-oauth2` | **Date**: 2026-02-25 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-mcp-oauth2/spec.md`

## Summary

Replace the shared API key authentication on MCP/GraphQL endpoints with per-user OAuth2 authentication using `league/oauth2-server-bundle`. This enables the existing LocaleService data access filters (clinic, section, workspace) to work correctly by authenticating real User entities instead of anonymous API key identities. The implementation follows the MCP Authorization specification (protocol revision 2025-11-25) with mandatory PKCE, Protected Resource Metadata (RFC 9728), Authorization Server Metadata (RFC 8414), and audience-bound tokens (RFC 8707). The consent flow delegates to the existing React frontend app.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| I. API Platform First | **JUSTIFIED BYPASS** | OAuth2 endpoints (`.well-known/*`, `/oauth/*`) are protocol-mandated JSON endpoints that cannot be modeled as API Platform resources. Custom controllers are required. Documented in research.md. |
| II. Domain-Grouped Organization | **PASS** | OAuth2 controllers grouped in `src/Controller/OAuth/`, entity in `src/Entity/`, services in `src/Service/`. Single domain concern (authentication). |
| III. Enum as Source of Truth | **PASS** | WorkspaceEnum already exists. No new enums needed — OAuth2 scope is a single string constant. |
| IV. YAGNI | **PASS** | Single scope `mcp:tools`, no admin UI, no granular permissions, no client credentials grant. DCR is MAY (deferred). Client ID Metadata Documents is SHOULD (included as minimal validation). |
| V. Lean MCP Layer | **PASS** | No new MCP tools. OAuth2 is infrastructure that enables existing tools to work with real user context. |
| VI. Multi-Workspace Awareness | **PASS** | Consent page includes workspace/country/clinic selection. Token carries context defaults. Headers override defaults. |

**Structure Decision**: Web application pattern — Symfony API handles OAuth2 server logic, React frontend handles the consent UI page. OAuth controllers are in a separate `src/Controller/OAuth/` directory with their own route configuration (no `/custom` prefix) since `.well-known` and `/oauth/*` paths are protocol-mandated.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| Custom controllers (bypasses Principle I) | OAuth2 protocol mandates specific endpoint paths (`.well-known/oauth-authorization-server`, `/oauth/authorize`, `/oauth/token`) with specific JSON schemas per RFCs 8414, 9728, 6749. These are not REST resources. | API Platform custom operations could theoretically serve JSON, but the endpoints don't map to any entity CRUD pattern and the bundle provides its own token controller. |
| `OAuth2AuthorizationContext` entity (new table) | Stores workspace/country/clinic defaults selected during consent. These must persist across the auth code → token exchange. | Encoding in the auth code itself would require modifying the bundle's auth code entity. A separate lookup table is cleaner and avoids forking the bundle. |
